# Plan Approval 标准（PA-01）

> 版本：v7.0
> 更新：2026-03-18
> 适用：mode:"plan" 的 Agent（@ui-designer）写 Plan、Lead 审 Plan 时必读
>
> **PA-01：写 Plan 前必读本文件。审 Plan 前必读本文件。不读不写，不读不审。**

---

## 总则

- **v7.0 分级授权**：仅 @ui-designer 使用 mode:"plan"。@coder-be/@coder-fe 为 auto 模式。验证类 Agent 为 auto 模式
- Plan 仍适用于 @ui-designer 的设计提案阶段，且必须包含可视化结构图
- Plan 不是待办清单，是**技术设计文档**。必须证明 Agent 真正理解了现有代码、想清楚了方案
- Plan 模板按任务级别分档（铁律 TG-01）：S 级 3 章、M 级 6 章、L/XL 级 9 章
- 通过线：加权总分 ≥ 3.5 / 5。任一维度 = 0 → 直接退回

### Agent Plan 的本质（皇上钦定 2026-02-17）

**Agent 的 Plan 不是 Lead 计划的复读机，而是该工位专家的施工方案。**

Lead 的计划定义"做什么"（目标、范围、退出条件）。Agent 的 Plan 回答"我作为这个领域的专家，打算怎么做好"。两层各有职责，不重叠。

**以项目核心为中心**：Agent 思考和优化方案时，必须围绕项目核心目标（见项目 CLAUDE.md `项目核心`），而非在技术真空中做活。每个决策要回答"这样做是否服务于项目核心"。

**Agent Plan 必须包含的四个部分**：

| # | 部分 | 说明 | 反面（禁止） |
|---|------|------|-------------|
| 1 | **我理解的任务** | 用自己的专业视角复述，不是抄 Lead 的原文 | 原样复制 Lead 的描述 |
| 2 | **我的执行策略和理由** | 执行顺序、为什么这样排序、哪些可以合并、哪些必须分开 | "按 Lead 给的顺序执行" |
| 3 | **我发现的风险和疑问** | 技术风险、边界情况、与项目核心的潜在冲突、Lead 可能没考虑到的点 | "没有风险" |
| 4 | **我建议的改进**（如有） | 更好的实现方式、Lead 方案的潜在问题、对项目核心更有利的替代方案 | 沉默不言（发现了问题但不说） |

**Lead 审批时的关注点**：不是检查"Agent 有没有抄对"，而是审核"Agent 的专业判断是否合理、是否服务项目核心、有没有发现我遗漏的问题"。

## 〇、Plan 模板分级（铁律 TG-01 联动）

Lead spawn 时在 prompt 中标注任务级别（S/M/L/XL），Agent 按对应级别写 Plan。

| 任务级别 | Plan 模板 | 必需章节 | 通过线 |
|----------|----------|----------|--------|
| **S 小修** | 3 章精简 | 第 0、1、3a 章 | 加权总分 ≥ 3.0 / 5 |
| **M 中改** | 6 章标准 | 第 0、1、2、3、5a、8 章 | 加权总分 ≥ 3.5 / 5 |
| **L/XL** | 9 章完整 | 全部 9 章 | 加权总分 ≥ 3.5 / 5 |

### S 级 Plan（3 章精简）

```
第 0 章：前置调查证据（≥1 文件引用）
第 1 章：任务复述（一句话目标 + 退出条件）
第 3a 章：逐步变更（每步 ≤30 行 + 验证方式）
```

量化门槛（S 级）：≥1 文件引用、每步 ≤30 行、≥1 验证命令。

### M 级 Plan（6 章标准）

```
第 0 章：前置调查证据（≥2 文件引用）
第 1 章：任务复述与项目核心对齐
第 2 章：现有代码分析
第 3 章：方案设计（3a 逐步变更 + 3b 关键决策）
第 5a 章：风险表（≥2 项 + 缓解措施）
第 8 章：自我批评
```

量化门槛（M 级）：≥2 文件引用、每步 ≤30 行、≥1 决策三问、≥2 风险项、≥1 自我批评。

### L/XL 级 Plan（9 章完整）

见下文完整 9 章模板。

## 一、Plan 强制模板（9 章 · L/XL 级完整版）

L/XL 级任务的 Agent Plan 必须按以下 9 章结构书写。**缺少任何章节 → 退回重写。**

### 第 0 章：前置调查证据

- 已读文件：`file:lines` — 读到的关键信息（≥ 3 个文件，不含契约）
- 已读历史错误：`.team-logs/` 中与本任务相关的条目（无相关条目须注明"已查无相关"）
- 已读契约：`contracts/` 中相关类型定义

### 第 1 章：任务复述与项目核心对齐

- 项目核心（引用项目 CLAUDE.md 中的 `项目核心` 定义）
- 确定性目标（一句话，用自己的专业视角复述）
- 本任务与项目核心的关系（这个任务如何服务于项目核心？）
- 退出条件（逐条列出，与 Lead spawn 中的退出条件 1:1 对应）

### 第 2 章：现有代码分析

- 当前实现：`file:line` — 现在是怎么做的（引用代码片段）
- 现有模式：项目中类似功能的实现模式 — 我要遵循哪个
- 上下游依赖：改 A → 影响 B, C（调用链列出）

### 第 3 章：方案设计

#### 3a. 逐步变更

每步 ≤ 30 行代码变更，超出必须拆分。每步必须按以下格式写：

| 字段 | 要求 |
|------|------|
| **现状** | `file:line_range` — 引用现有代码片段，证明读过 |
| **改为** | 写出具体代码结构、函数签名、数据结构（不是自然语言描述） |
| **理由** | 遵循 `file:line` 的现有模式 |
| **不变量** | 改完后哪些系统性质必须仍然成立 |
| **失败模式** | 这步具体可能怎么出错 |
| **验证** | 具体可执行的命令或操作（不是"检查是否正常"） |
| **回退** | 如何只撤销这步 |

#### 3b. 关键决策

每个技术选型 / 设计决策必须附带：

| 字段 | 要求 |
|------|------|
| **选了什么** | A |
| **做了（选 A）** | 影响分析 |
| **不做（不选 A）** | 代价分析 |
| **替代（选 B）** | 利弊对比 |
| **反驳自己** | 选 A 最大的风险是什么？为什么仍然选？ |
| **备选方案** | 如果 A 失败，退而用什么？ |

#### 3c. 错误路径

- ≥ 2 个失败场景（具体描述，不是"可能出错"）
- 边界情况枚举

### 第 4 章：多维场景矩阵

7 维必须逐维回答。不适用标 "N/A + 理由"，但不可全标 N/A，**至少 5 维有实质分析**：

| 维度 | 必须回答 |
|------|----------|
| **用户维度** | 新用户首次使用 vs 老用户已有数据 — 体验分别如何？ |
| **数据维度** | 空状态 / 单条 / 大量 / 损坏数据 — 分别怎么处理？ |
| **时序维度** | 首次加载 / 反复操作 / 重启恢复 / 长时间运行 — 有无退化？ |
| **并发维度** | 快速重复操作 / 多窗口 / 后台任务未完成时新操作 — 会否冲突？ |
| **失败维度** | 网络断 / 后端错误 / 数据格式不符 / 权限不足 — 用户看到什么？ |
| **集成维度** | 改了 A 模块，B/C 的哪些功能受影响？事件链、状态传播、UI 联动？ |
| **性能维度** | 时间复杂度？内存占用？渲染次数？包体积影响？ |

### 第 5 章：风险与事前验尸

#### 5a. 风险表

每项必须有具体缓解方案：

| 风险 | 概率(高/中/低) | 影响(高/中/低) | 缓解措施 |
|------|---------------|---------------|----------|

#### 5b. 事前验尸

假设 plan 已执行且失败，最可能的 3 个原因（倒推思维）：

1. 原因 — 为什么可能发生 — 预防措施
2. 原因 — 为什么可能发生 — 预防措施
3. 原因 — 为什么可能发生 — 预防措施

### 第 6 章：领域专业项

按角色回答对应专属清单（见下文第三节）。每项必须回答，不适用须说明理由。

### 第 7 章：不变量声明

列出本次修改前后必须保持不变的系统核心性质：

- INV-1: 描述 — 验证方法
- INV-2: 描述 — 验证方法
- ...

### 第 8 章：自我批评

- 本 plan 最弱的部分是什么？（必须具体，**不可写"没有弱点"**）
- 最可能被退回的点是什么？
- 补救建议：如何加强弱点？

## 二、量化门槛（L/XL 级完整版）

> S/M 级的量化门槛见「〇、Plan 模板分级」对应小节。以下为 L/XL 级完整门槛。

不满足以下**任一项** → **直接退回**，不进入内容审核：

| 维度 | 最低要求 |
|------|----------|
| 项目核心引用 | 第 1 章必须引用项目核心并分析关联 |
| 前置调查文件数 | ≥ 3（不含契约） |
| 每步变更最大行数 | ≤ 30 |
| 代码引用密度 | 每 100 字 plan 至少 1 处 `file:line` 引用 |
| 决策三问 | 每个技术选型 / 设计决策 1 组（含反驳） |
| 错误路径 | ≥ 2 个失败场景 |
| 场景矩阵 | 7 维中至少 5 维有实质分析 |
| 事前验尸 | ≥ 3 条 |
| 自我批评 | ≥ 1 条实质弱点 + 补救（不可写"无弱点"） |

## 三、角色专属技术审视项

### @coder-be (Rust/Tauri)

Plan 中必须回答：

| # | 审视项 |
|---|--------|
| BE-1 | 新 struct 所有权设计：哪些字段 owned、哪些 borrowed？为什么？ |
| BE-2 | 错误处理策略：thiserror / anyhow / 自定义？错误码与契约对应关系？ |
| BE-3 | 异步考虑：有无阻塞调用？需要 spawn_blocking？ |
| BE-4 | 序列化：serde derive 够还是需要自定义？字段命名策略？ |
| BE-5 | Tauri 特有：command 注册方式、capability 声明、State 管理、event emit 格式 |
| BE-6 | 线程安全：跨线程共享数据方案（Arc\<Mutex\>? RwLock? 无锁?） |

### @coder-fe (TypeScript/React)

Plan 中必须回答：

| # | 审视项 |
|---|--------|
| FE-1 | 组件 props 接口（写出来）。受控 vs 非受控？compound component？ |
| FE-2 | 重渲染分析：什么触发重渲染？需要 useMemo / useCallback？父组件更新影响？ |
| FE-3 | 状态归属：local state / zustand store / props？为什么？ |
| FE-4 | 副作用清理：useEffect cleanup、事件监听 remove、定时器 clear |
| FE-5 | 类型安全：泛型设计、联合类型、类型守卫 |
| FE-6 | i18n：新增 key 列表（zh-CN + en-US 都列出） |
| FE-7 | Mock 策略：mock 位置、联调切换点、mock 清理方式 |

### @ui-designer

Plan 中必须回答：

| # | 审视项 |
|---|--------|
| UI-1 | 交互状态全枚举：default / hover / active / focus / disabled / loading / error / empty / skeleton |
| UI-2 | 所有颜色/间距/圆角引用 design token 名（不可写具体值） |
| UI-3 | 视觉层级：信息优先级排序 → 视觉权重分配 |
| UI-4 | 响应式：不同容器宽度的布局变化（如适用） |
| UI-5 | 动效：GPU 加速？duration / easing 引用 token？ |

### @reviewer / @cross-checker / @e2e-verifier / @debugger

验证类 Agent 不写 Plan，但需了解本标准以验证：
- 实现是否符合 Plan 承诺的方案
- Plan 中声明的不变量是否被破坏
- Plan 中识别的风险是否已被缓解

## 四、9 维评分标准

Lead 审核时逐维打分（0-5），加权计算总分：

| 维度 | 权重 | 0 分 | 3 分 | 5 分 |
|------|------|------|------|------|
| 项目核心对齐 | 10% | 未提及项目核心 | 提及但分析浅 | 每个关键决策都回扣项目核心 |
| 专家独立思考 | 10% | 原样复制 Lead 指令 | 有自己的执行策略 | 策略有理由 + 发现 Lead 遗漏 + 提出改进 |
| 现有代码理解 | 10% | 没读代码 | 引用了但理解浅 | 精确引用 + 识别模式 + 上下游 |
| 方案颗粒度 | 15% | "修改 X 组件" | 写了函数名 | 函数签名 + 数据结构 + 代码片段 |
| 多维场景 | 15% | 只有正向路径 | 覆盖 3 维 | 7 维矩阵实质覆盖 |
| 风险 + 事前验尸 | 15% | "可能有风险" | 风险 + 缓解 | 风险 + 缓解 + 事前验尸 3 条 |
| 专业深度 | 10% | 通用描述 | 回答了角色专属项 | 全部回答 + 独到见解 |
| 决策质量 | 10% | 没说为什么 | 有三问 | 三问 + 反驳自己 + 备选 |
| 自我批评 | 5% | "没弱点" | 找到 1 弱点 | 弱点 + 补救方案 |

**通过线：加权总分 ≥ 3.5 / 5。任一维度 = 0 → 直接退回，不看总分。**
